# HARNESS.md — 하네스 엔지니어링 플로우 가이드

> 이 문서는 "대화 기반 요청 → Claude 개발 → 테스트" 플로우를
> 어떻게 실행하는지 설명합니다.

---

## 하네스 엔지니어링이란?

하네스(Harness)는 원래 말이나 장비에 채우는 **안전 장치 · 마구**를 의미합니다.
AI 보조 개발에서 하네스 엔지니어링이란:

> AI가 코드를 생성할 때 잘못된 방향으로 가지 않도록 붙잡아주는
> **규칙, 제약, 자동화 안전망**의 총체

단순히 Claude에게 "이거 만들어줘"라고 하는 것이 아니라,
**어떻게 만들어야 하는지 사전에 정의하고** → AI가 그 안에서 작동하게 합니다.

## 이 프로젝트의 하네스 구조

```
┌─────────────────────────────────────┐
│           하네스 레이어              │
│                                     │
│  CLAUDE.md      ← AI 행동 지침      │
│  CONVENTIONS.md ← 코드 컨벤션       │
│  TESTING.md     ← 테스트 기준       │
│                                     │
│  .github/
│    ├── PULL_REQUEST_TEMPLATE.md     │
│    ├── ISSUE_TEMPLATE/              │
│    └── workflows/
│          ├── ci.yml   ← 자동 검증   │
│          └── e2e.yml  ← E2E 테스트  │
└─────────────────────────────────────┘
```

## 개발 플로우 (단계별)

### Step 1. 이슈 생성

GitHub에서 기능 요청 또는 버그 이슈를 생성합니다.
이슈 템플릿의 **"Claude에게 전달할 요청 초안"** 항목을 작성합니다.

```
# 좋은 요청 예시
"RSVP 제출 폼을 만들어줘.

요구사항:
- 이름(필수), 연락처(필수), 참석 인원(1-10명) 필드
- 참석 여부 선택 (참석/불참)
- 유효성 검사 실패 시 인라인 에러 메시지

참고:
- CLAUDE.md의 컴포넌트 구조 준수
- CONVENTIONS.md의 네이밍 규칙 준수
- TESTING.md 기준으로 테스트 파일 함께 작성해줘"
```

### Step 2. 브랜치 생성

```bash
git checkout -b feat/12-rsvp-form
```

### Step 3. Claude에게 요청

이슈에서 작성한 요청을 Claude에게 전달합니다.
Claude는 CLAUDE.md, CONVENTIONS.md, TESTING.md를 기반으로 코드를 생성합니다.

### Step 4. 코드 적용 & 로컬 확인

```bash
# Claude가 생성한 파일 적용
pnpm test:watch     # 테스트 통과 확인
pnpm dev            # 브라우저에서 모바일 시뮬레이터로 확인
```

### Step 5. 커밋 & Push

```bash
git add .
git commit -m "feat(rsvp): 참석 여부 제출 폼 추가

- 이름, 연락처, 참석 인원 입력 필드
- 유효성 검사 포함

Closes #12"

git push origin feat/12-rsvp-form
```

### Step 6. PR 생성

PR 템플릿을 채우고 PR을 생성합니다.
Vercel이 자동으로 Preview URL을 댓글로 달아줍니다.

### Step 7. CI 자동 검증

PR이 생성되면 GitHub Actions가 자동으로:

1. ESLint + TypeScript 타입 체크
2. Jest 테스트 실행
3. Next.js 빌드 검증

모든 체크가 녹색이면 PR 머지 가능 상태가 됩니다.

### Step 8. E2E 테스트

Vercel Preview 배포 완료 후 Playwright E2E 테스트가 자동 실행됩니다.

### Step 9. 머지 & 배포

모든 체크 통과 → PR 머지 → Vercel 자동 프로덕션 배포

## 하네스 효과

| 문제 상황                | 하네스 없이                    | 하네스 있을 때                    |
| ------------------------ | ------------------------------ | --------------------------------- |
| Claude가 `any` 타입 사용 | 빌드는 되지만 런타임 오류 가능 | CLAUDE.md 금지 규칙 → 재요청      |
| 커밋 메시지 중구난방     | 이력 추적 어려움               | CONVENTIONS.md 기준으로 수정 요청 |
| 테스트 없이 배포         | 회귀 버그 발견 불가            | CI 게이트에서 차단                |
| 브랜치 전략 미준수       | main 직접 push                 | 브랜치 보호 규칙으로 강제 차단    |

## 자주 쓰는 Claude 요청 패턴

```
# 새 컴포넌트
"[컴포넌트명] 컴포넌트를 만들어줘.
[요구사항 목록]
CLAUDE.md와 CONVENTIONS.md 기준으로,
TESTING.md에 맞는 테스트 파일도 함께 작성해줘."

# 버그 수정
"[파일경로]의 [문제]를 수정해줘.
에러 메시지: [에러]
CONVENTIONS.md 컨벤션 유지해줘."

# 리팩토링
"[파일경로]를 CONVENTIONS.md 기준으로 리팩토링해줘.
기능은 변경하지 말고 코드 품질만 개선해줘.
기존 테스트가 통과하는지 확인해줘."
```
